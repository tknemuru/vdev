# レビュー統合アクションプラン（ラウンド 1）

## 判定
- **Status**: Request Changes
- **P0 件数**: 1 件
- **P1 件数**: 5 件（アクション対象: 5 件）
- **P2 件数**: 3 件（記録のみ）

## 必須対応 (P0)

**[P0-1] G1・G2 の達成確認方法がトートロジーに該当する**
- **出典**: Approach
- **内容**: G1 の Verification は「3層モデルの説明が含まれていること」「判断基準が示されていること」等、実装完了と同義の記述であり、品質や効果を検証していない。G2 も同様に「〜が記載されていること」の列挙に終始している。これは検証項目で P0 として指摘すべきと定義されている「実装完了と同義になるトートロジー的な確認方法」に該当する。
- **修正方針**: 達成確認方法を「実行可能な確認手順」に変更する。例えば G1 は「改定後の CLAUDE.md を用いて AI に未知のプロジェクトの System Overview Docs を起草させ、技術的事実（エンドポイント一覧等）がドキュメントに含まれないことを確認する」、G2 は「改定後の `/rfc` を実行し、Step 5 で AI が docs の記載とコードの矛盾を検出・報告するケースを1件以上確認する」等、成果物の存在ではなく行動変容を検証する手順とする。

## 推奨対応 (P1) — 上位5件

**[P1-1] 3層モデルの CLAUDE.md への記載がスコープ過大である可能性**
- **出典**: Approach
- **内容**: 3層情報アーキテクチャの定義テーブル全体を CLAUDE.md に記載する設計であるが、CLAUDE.md は AI への指示ファイルであり、情報アーキテクチャの理論的背景を説明する場ではない。AI が必要とするのは「docs に何を書き、何を書かないか」の判断基準であり、3層モデル自体の説明は docs のガイドラインとして機能すれば十分である。
- **修正方針**: 3層モデルの定義テーブルは簡潔な1文の要約に置き換え、「記載内容の判断基準」テーブルと System Overview Documents テーブルに情報を集約する。CLAUDE.md の記載量を最小限にとどめ、AI が実際に従うべきルールを明瞭にする。

**[P1-2] IEEE 1016 への言及除去の根拠が不十分であり、設計根拠が空白になる**
- **出典**: Approach, Security
- **内容**: IEEE 1016 の3ビューポイント（Structure / Data / Interface）は3ドキュメント構成の設計根拠であり、3層モデルの導入はドキュメントの記載内容ガイドラインの変更であって、ドキュメント構成自体の変更ではない。設計根拠を除去すると、なぜ3ドキュメントに分けるのかの根拠が失われる。また、3層モデル自体の理論的根拠がレポートへの参照のみに依存しており、将来レポートが陳腐化した場合に採用根拠が失われる可能性がある。
- **修正方針**: IEEE 1016 への言及は維持するか、3ドキュメント構成の根拠を別の形で明記する。例えば「IEEE 1016 の3ビューポイントに対応する3ドキュメント構成は維持し、各ドキュメントの記載内容をドメイン知識中心に改定する」とする。

**[P1-3] architecture.md の記載内容から「ディレクトリ構成」を除去した判断の再考**
- **出典**: Quality
- **内容**: 「ディレクトリ構成」はコードから導出可能であるが、大規模リポジトリではディレクトリ構造の全体像を素早く把握するための「構造的オリエンテーション」としての価値がある。完全な除去ではなく、「主要ディレクトリの責務概要」のような形でオリエンテーション目的の記載を残すことで、AI が調査の起点を得やすくなる。
- **修正方針**: `architecture.md` の記載内容に「主要ディレクトリの責務概要」を追加し、技術的事実としてのディレクトリ構成ではなく構造的オリエンテーションとしての位置づけを明確にする。

**[P1-4] `/rfc` Step 5b の調査指示と他ステップとの一貫性確認**
- **出典**: Quality
- **内容**: 現行の `/rfc` コマンドにおいて、Step 5 以外のステップ（例: Step 6 の起草指示）が System Overview Docs を技術的事実のソースとして暗黙に参照している箇所がないか確認されていない。Step 5 のみ改定しても他ステップに旧来の前提が残る場合、コマンド内で指示の一貫性が崩れる。
- **修正方針**: `adapters/claude/commands/rfc.md` の全ステップを確認し、Step 5 の方針変更と矛盾する記述がないことを RFC 内で明記する。矛盾があれば変更対象に追加する。

**[P1-5] csync 同期時の既存 docs との不整合に対する段階的移行ガイダンスの欠如**
- **出典**: Security, Quality
- **内容**: csync 同期後、各プロジェクトの既存 System Overview Docs が新ガイドラインと不整合になるが、「各プロジェクトで段階的に行う」とのみ記載されており、移行の優先度や手順の指針がない。AI が新旧混在の docs を参照した際の振る舞いが不確定になるリスクがある。
- **修正方針**: 7.4 に「既存 docs の改修は各プロジェクトの次回 RFC 起草時などの自然なタイミングで行う」等の移行タイミングの指針を1〜2文追記する。

## 記録のみ (P2)

- `/rfc` 以外のコマンドへの波及影響が未評価である。将来的に `/imp` 等の他コマンドでの影響を観察し、問題があれば個別に対応するとの方針を明記する程度でよい。（出典: Approach）
- 行動原則層（`.claude/rules/`）の運用ガイダンスが不在である。Non-Goals でスコープ外としているが、行動原則層の具体的な運用方法が別途定義されるまで活用が進まない可能性がある。（出典: Security）
- Step 5b の調査負荷増加に対するガードレールが欠如している。大規模リポジトリにおいて AI が広範なコード調査を行い、RFC 起草の時間・トークンコストが大幅に増加するリスクがある。調査範囲の目安がないと青天井になりうる。（出典: Quality）

## 記載内容の判断基準テーブルにおける境界ケースの明確化について

Quality P1-3 の指摘（記載内容の判断基準テーブルにおける境界ケースの明確化）は、P1-1（CLAUDE.md への記載スコープの見直し）の修正作業に内包される。P1-1 で判断基準テーブルに情報を集約する際に「コードに明示されない意思決定の背景・理由は docs に書く」旨の補足を追加することで対応する。

#### 指摘一覧

**P0**: 該当なし

**[P1-1] architecture.md の記載内容から「ディレクトリ構成」を除去した判断の再考**
- **対象セクション**: 5.2 変更1: CLAUDE.md の System Overview Documents セクション改定
- **内容**: 「ディレクトリ構成」はコードから `tree` コマンド等で導出可能であるが、大規模リポジトリではディレクトリ構造の全体像を素早く把握するための「構造的オリエンテーション」としての価値がある。完全な除去ではなく、「主要ディレクトリの責務概要」のような形でオリエンテーション目的の記載を残すことで、AI が調査の起点を得やすくなる。
- **修正の期待値**: `architecture.md` の記載内容に「主要ディレクトリの責務概要」を追加し、技術的事実としてのディレクトリ構成ではなく構造的オリエンテーションとしての位置づけを明確にする。

**[P1-2] `/rfc` Step 5b の調査指示に対する既存コードベースパターンとの一貫性確認**
- **対象セクション**: 5.3 変更2: `/rfc` Step 5 の調査指示改定
- **内容**: 現行の `/rfc` コマンドにおいて、Step 5 以外のステップ（例: Step 6 の起草指示）が System Overview Docs を技術的事実のソースとして暗黙に参照している箇所がないか確認されていない。Step 5 のみ改定しても他ステップに旧来の前提が残る場合、コマンド内で指示の一貫性が崩れる。
- **修正の期待値**: `adapters/claude/commands/rfc.md` の全ステップを確認し、Step 5 の方針変更と矛盾する記述がないことを RFC 内で明記する、または矛盾があれば変更対象に追加する。

**[P1-3] 記載内容の判断基準テーブルにおける境界ケースの明確化**
- **対象セクション**: 5.2 変更1: 記載内容の判断基準
- **内容**: 「コードから導出できるか？」という基準において、「技術選定の理由」のように docs に記載する項目でありながらコードのコメントやコミット履歴から部分的に導出可能なケースの扱いが曖昧である。判断基準の「できる/できない」は二値的だが、実際には程度の問題が存在する。
- **修正の期待値**: 判断基準の補足として「コードに明示されない意思決定の背景・理由は docs に書く」旨を1文追加し、境界ケースの判断指針を示す。

**[P2-1] 5b の調査負荷増加に対するガードレールの欠如**
- **対象セクション**: 7.2 スケーラビリティとパフォーマンス
- **内容**: Step 5b で「コードベースを直接調査」する範囲にガードレールがない。大規模リポジトリにおいて AI が広範なコード調査を行い、RFC 起草の時間・トークンコストが大幅に増加するリスクがある。7.2 では「コストのほうが総合的には小さい」と記載されているが、調査範囲の目安がないと青天井になりうる。

**[P2-2] csync 同期後の既存 docs との整合性に関する運用ガイダンスの不足**
- **対象セクション**: 7.4 マイグレーションと後方互換性
- **内容**: 新ガイドライン同期後、既存 docs に技術的事実が残存する過渡期が長期化した場合、docs の信頼性が不明確な状態が続く。Step 5 の注意書きで部分的に対処しているが、各プロジェクトで「いつまでに」「どの優先度で」docs を改修すべきかの指針がない。本 RFC のスコープ外であるが、将来的に運用ガイダンスの策定が望ましい。
